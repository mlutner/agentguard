## What the Panel Found

### Strengths (what to protect)

1. **The @guard decorator is genuinely elegant** — Sentry-style bottoms-up adoption works because the integration cost is one line of code (Champion, Data Scientist, Incumbent all acknowledged this)
2. **Real pain point** — agent loops and runaway costs are universal, documented production problems (Champion: 8/10, First Customer: 7/10)
3. **Proven distribution model** — free SDK → cloud upsell mirrors Sentry ($200M+ ARR), Datadog, PostHog (Data Scientist)

### Weaknesses (what to fix)

| # | Weakness | Who Said It | Severity | Actionable? |
|---|----------|-------------|----------|-------------|
| 1 | **Zero users = zero trust** — no one will let a pre-v1.0 SDK gatekeep production agents | First Customer, Analyst, Incumbent | Critical | Yes |
| 2 | **LangSmith clones this in one sprint** — LangChain has 100K+ devs, tracing data, and distribution moat | Killer, Incumbent | Critical | Yes |
| 3 | **Decorator is structurally too soft** — doesn't survive cross-process escapes, self-modifying agents, or async race conditions | Outsider, First Customer | High | Yes |
| 4 | **No empirical FPR/FNR data** — false positive in production destroys trust irreversibly | Data Scientist | High | Yes |
| 5 | **Drift detection is hard to define universally** — might bloat an otherwise focused SDK | Champion | Medium | Yes |

### Unanswered Questions (from panelists)

1. How does the SDK calculate costs across different LLM providers without fragile integrations?
2. What telemetry leaves the VPC for cloud-side drift analysis?
3. Fail-closed vs fail-open behavior — what happens when AgentGuard itself crashes?
4. How do you handle agents that spawn sub-workers or rewrite their own execution context?
5. Who is the economic buyer — the developer or the engineering manager?

## Action Plan (derived from panel feedback)

### Priority 1: Prove it works (addresses #1, #4)

**Problem:** Zero users and no empirical data means the safety claim is unverifiable.

**Actions:**
- [ ] Run benchmarks: n≥384 agent runs across LangChain, CrewAI, and AutoGen (Data Scientist's H0 methodology)
- [ ] Measure and publish: false positive rate, false negative rate, latency overhead per check
- [ ] Create reproducible adversarial scenarios: infinite loop, cost explosion, topic drift, agent stall
- [ ] Add benchmark results to README with methodology section
- [ ] Get 10 developers testing within 2 weeks (even unpaid design partners)

### Priority 2: Harden the architecture (addresses #3)

**Problem:** A Python decorator can be bypassed by async race conditions, sub-workers, and self-modifying code.

**Actions:**
- [ ] Add async-native guard variant (not just wrapping sync decorator over async)
- [ ] Add sub-agent propagation: when a guarded agent spawns children, guard propagates
- [ ] Add fail-open/fail-closed configuration: `@guard(on_error="open")` vs `@guard(on_error="closed")`
- [ ] Document explicitly what the guard does NOT protect against (scope boundaries)
- [ ] Add process-level heartbeat (not just function-level) for long-running agents

### Priority 3: Build the moat before LangSmith copies it (addresses #2)

**Problem:** LangChain can ship loop/cost/drift features in LangSmith within one quarter.

**Actions:**
- [ ] Ship to PyPI immediately — first-mover advantage on the specific "guardrail decorator" pattern
- [ ] Focus on framework-AGNOSTIC as the differentiator (LangSmith only works with LangChain)
- [ ] Add adapters for AutoGen, DSPy, and Anthropic's tool-use patterns (not just LangGraph/CrewAI)
- [ ] Build the calibration dataset: every guard intervention is training data LangSmith doesn't have
- [ ] Consider: is this an acquisition target for LangChain, not a competitor? Position accordingly.

### Priority 4: Simplify drift detection (addresses #5)

**Problem:** Universal drift detection is an unsolved research problem. Don't let scope creep bloat the SDK.

**Actions:**
- [ ] Make drift detection opt-in (not default) — the other 3 checks are the core value
- [ ] Ship with simple keyword-based drift first, embeddings as opt-in upgrade
- [ ] Document the drift threshold calibration process clearly
- [ ] Consider removing drift from v1 entirely and shipping as a v1.1 add-on

### Priority 5: Answer the economic buyer question (addresses unanswered #5)

**Problem:** The panel couldn't identify who actually pays for this.

**Actions:**
- [ ] Define the buyer: Engineering Manager who got a $2,000 surprise API bill
- [ ] Add cost-saved metrics to the guard output (not just "stopped" but "saved $X")
- [ ] Build the ROI story: one prevented runaway loop pays for a year of cloud subscription
- [ ] Position cloud tier around team-level controls, not individual developer features
